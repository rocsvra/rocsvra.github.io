Redis(4)：数据持久化

发表于 2023-02-28 更新于 2023-09-11 分类于 数据库 ， Redis

redis提供了两种持久化方式：

RDB（Redis DataBase），在不同的时间点，将redis存储的数据生成快照并存储到磁盘等介质上；
AOF（Append Only File），将redis执行过的所有写指令记录下来，在下次redis重启时，将这些指令重复执行一遍，就可以实现数据恢复；

RDB和AOF两种方式可以同时使用，如果这种情况下redis重启，则优先采用AOF进行数据恢复，因为这种方式数据恢复完整度更高。官方的建议就是两个同时使用，这样可以提供更可靠的持久化方案。
如果没有数据持久化需求，可以完全关闭RDB和AOF，redis将变成一个纯内存数据库，就像memcache。

RDB

先将数据写入一个临时文件，待持久化结束，用这个临时文件替换上次的持久化文件。我们随时可以进行备份，快照文件总是完整可用。
redis会单独fork一个子进程来进行持久化，主进程是不会进行任何IO操作的，确保了redis极高的性能。
如果需要进行大规模数据恢复，且数据恢复的完整性不是非常敏感，RDB比AOF更加高效；
如果对数据完整性敏感，即使每半小时持久化一次，redis故障时，仍会有近半小时的数据丢失，这时我们选择AOF。

AOF

如果有写操作，就会被redis追加到AOF文件的末尾；
默认AOF持久化策略是每秒fsync一次（fsync是指把缓存中的写指令记录到磁盘中）,即使故障只会丢失近一秒的数据；
同样的数据规模，AOF文件要比RDB文件的体积大，AOF方式的恢复速度也要慢于RDB方式；
